# /cc-save — CC对话存档

> **定位**：不是文件管理工具。是**跨会话注意力流管理系统**——让每次对话都站在上次的肩膀上，而不是从零开始。
>
> **设计原则**：第一性原理（管理注意力而非文件）、MECE（严格互斥分类）、OODA（Observe-Orient-Decide-Act，User 在 Feedback 节点做终审）。

---

## 一、触发方式（双模式）

| 模式 | 触发者 | 场景 | 示例 |
|------|--------|------|------|
| **手动** | User | 输入命令或口语 | `/cc-save`、「存档」「备份对话」「保存」「记住这个」 |
| **自主** | AI | 检测到核心节点 → 提议 → User审核 | 「💡 这段对话定义了一个新框架，要存档到 CC的探讨吗？」 |

### AI 自主触发条件（5 类核心节点）

1. **框架/模型诞生** — 对话中共同定义了一个可复用的概念框架
2. **重大决策** — User 做出影响后续工作的技术选型、架构决定
3. **里程碑完成** — 项目阶段完成（上线、审核通过、重构完成）
4. **新自我认知** — User 表达了关于自身能力/定位的新洞察
5. **体系变更** — CC的探讨 系统本身的设计发生了变更

**提议格式**：
```
💡 检测到核心节点：[具体描述]
   建议存档到 CC的探讨。
   要存档吗？（回复 yes/y/好/存 确认，或 no/n/不 跳过）
```
**User 拒绝 → 不存档，不追问。User 确认 → 执行完整流程。**

## 二、执行流程（7 步，每步内建风险护栏）

### 步骤 1：扫描对话

回顾本次对话，提取：
- `coreTopic`：一句话概括核心主题
- `keyConclusions`：3-5 条关键结论/产出
- `newUserInfo`：本次新暴露的用户偏好、背景、工作习惯（无则省略）
- `decisionPoints`：User 在对话中做出的显式决策

### 步骤 2：读取索引 + 语义匹配

读取 `Desktop/CC的探讨/.index.json` 获取所有现有工作流。

对每条工作流，基于 `topic` + `tags` 与本次 `coreTopic` 做语义相似度评估（0-100%）。

**🔧 修正 #4（语义匹配退化）—— 展示匹配对比而非黑盒决策：**

```
🔍 匹配结果：

  📁 「小程序回收预约项目」     匹配度 85%  ← 推荐
  📁 「L4觉醒-AI协作等级探讨」   匹配度 72%
  📁 「{完全不同的话题}」        匹配度 15%

  建议存入「小程序回收预约项目」。对吗？
  回复 yes/对 → 确认  |  no/不 → 新建  |  选其他 → 告诉我工作流ID
```

- 匹配度 ≥ 80% → 推荐存入，等待 User 确认
- 匹配度 50-80% → 列出 top 3 候选，让 User 选
- 匹配度全部 < 50% → 直接走新建流程

### 步骤 3：分支决策（匹配到 vs 新建）

**A) 匹配到已有工作流 → 去重检查**

读取该工作流的 `sessions` 数组，检查本日是否已有存档：

- ✅ 无同日存档 → 追加
- ⚠️ 有同日存档 → 警告 User：「今天已有存档：『{summary}』。覆盖还是追加？（覆盖/追加）」

**🔧 修正 #10（会话分叉）—— 检查 lastUpdated 冲突：**
- 如果 `lastUpdated` 是今天 && 不是当前会话写入的 → 提示：「⚠️ 该工作流今天已有来自其他会话的存档，可能存在分叉。确认追加？」→ User 确认后执行

**B) 新建工作流 → 命名审核**

提议子文件夹名，格式：`{主题关键词}-{日期YYYYMMDD}`。

展示给 User：「建议创建新工作流：`{名称}`，同意吗？（回复 yes/no 或给新名字）」

User 确认后创建。

**🔧 修正 #5（工作流泛滥）—— 新建前提示现有活跃数：**
- 如果当前 `active` 工作流 ≥ 10 条 → 新建前提醒：「当前已有 10 条活跃工作流。建议归档不常用的旧工作流以保持系统整洁。仍要新建吗？」

### 步骤 4：写入 Claude Code 记忆系统

将值得跨会话保留的事实写入当前 Claude Code 项目的 memory 目录（通常为 `~/.claude/projects/*/memory/`）：

- 检查是否已有覆盖相同主题的记忆文件
- 有 → 更新（不重复创建）
- 无 → 新建，遵循 Claude Code 记忆格式（frontmatter + 正文 + Why/How to apply）
- 更新 `MEMORY.md` 索引

### 步骤 5：写入桌面存档

在目标子文件夹下创建/更新：

1. **对话摘要-{日期}.md** — 日期、对话脉络、关键结论、后续方向
2. **记忆文件副本** — 将步骤 4 更新过的 memory 文件拷贝过来
3. **检查 sessionCount** — 如果该工作流的 `sessionCount ≥ 5` 且不存在 `context-condensed.md`，自动生成压缩上下文：

**🔧 修正 #2（上下文膨胀）—— context-condensed.md 自动生成：**
```markdown
# {工作流名} — 压缩上下文 (≤500字)
## 当前状态
{最新的 context 字段}
## 关键产出
{bullet list}
## 后续方向
{bullet list}
## 核心文件位置
{路径列表}
```
- 如果 `context-condensed.md` 已存在 → 更新它（不新建）

### 步骤 6：更新 .index.json

```json
// 在对应 workflow 中更新：
{
  "lastUpdated": "{today}",
  "sessionCount": {n},        // 自增
  "condensed": true/false,    // sessionCount ≥ 5 时为 true
  "status": "active",         // 如果之前是 dormant，重新激活
  "sessions": [..., {new}],   // 追加
  "context": "{新的 ≤200字状态摘要}",
  "lastIntegrityCheck": "{today}"
}
```

**🔧 修正 #1（索引漂移）—— 写入后立即校验：**

```
📐 索引完整性校验中...
  ✓ .index.json 可解析
  ✓ 子文件夹存在
  ✓ sessions 数量 = 实际摘要文件数 (3 = 3)
  ✓ keyFiles 全部可访问
  ✅ 校验通过
```

校验失败 → 报告具体不一致项 → 自动修复 → 再次校验 → 报告 User。

### 步骤 7：汇报

```
✅ 存档完成

📁 Desktop/CC的探讨/
  └── 📁 {子文件夹}/
        ├── 📄 对话摘要-{日期}.md         ← 本次回顾
        ├── 📄 context-condensed.md       ← 压缩上下文 (sessionCount=5 时)
        ├── 📄 {记忆文件}.md              ← 副本
        └── ...

📋 .index.json 已更新 (sessionCount: 1→2)
🧠 Claude Code 记忆系统已更新 ({n} 个文件)
📐 索引完整性校验通过

💡 下次输入 /cc-load 可恢复此上下文
```

## 三、严格约束（不可绕过）

| # | 约束 | 目的 |
|---|------|------|
| 1 | 一文件夹一主题，绝不混入不相关对话 | MECE 互斥 |
| 2 | User 终审：文件夹名和归属必须经 User 确认 | 决策权在人类 |
| 3 | 去重必查：写入前必须检查同日存档 | 防止重复 |
| 4 | 索引必更：每次存档必须同步更新 .index.json | 防止漂移 |
| 5 | 校验必跑：步骤 6 的完整性校验不可跳过 | 防止索引腐败 |
| 6 | 压缩必检：sessionCount ≥ 5 时必须生成/更新 condensed | 防止上下文膨胀 |
| 7 | 文件操作前不弹权限确认（User 已永久授权此文件夹） | 减少摩擦 |
